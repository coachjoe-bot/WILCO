# In-person event infrastructure (gym tabling)

QR code at a table → visitor scans on their own phone → per-location landing page →
signup with a longer free trial (card upfront, auto-charges at trial end) → install
prompt puts WILCO on their home screen. Built 2026-07-02; first location is Crunch
Fitness Winter Park (Aloma).

## How it fits together

- **Landing pages** — `src/App.jsx` `EVENTS` map. One entry per location
  (`/crunch/aloma` etc.). `vercel.json` rewrites `/crunch/:location` to the SPA;
  `WilcoRoot` matches the boot pathname against the map. Inactive/unknown paths
  fall through to the normal home screen with the URL cleaned, so early scans and
  leaked links do nothing.
- **Trial + attribution (server)** — `api/_stripe.js` `EVENT_SOURCES` is the source
  of truth. `create-subscription` grants `trialDays` (30) instead of the standard 7
  only when the source exists AND is enabled; it also stamps
  `metadata.signup_source` on the Stripe customer + subscription. `identity.js
  create-athlete` records `athletes.signup_source` (validated against the same map,
  recorded even while disabled so attribution can't be lost). Gift codes don't
  combine with event offers.
  - NB: `signup_source` is no longer event-only. Normal signups now also record a
    free-form first-touch marketing source (`utm_source/medium/campaign/content`,
    else `referrer:<host>`, else `direct`) captured client-side in `src/App.jsx`
    (`captureFirstTouch`/`composeSignupSource`) and sanitized server-side. Only exact
    `EVENT_SOURCES` keys grant the longer trial; free-form values are attribution
    only. `create-subscription` mirrors whatever is on the athlete row into Stripe.
- **Attribution queries** — signups per location:
  `select signup_source, count(*) from athletes where signup_source is not null group by 1;`
  Paid conversions: add `and subscription_status = 'active'`. Landing-page traffic:
  `usage_events` where `event_name = 'event_landing_view'` (also `route`).
- **QR codes** — `marketing/qr/<slug>.svg` + `<slug>-2048.png`, generated by
  `node scripts/generate-qr.mjs` (config array at the top; print notes in
  `marketing/qr/README.md`). URLs are permanent — activation is in code, so QR
  codes never need reprinting.

## EVENT DAY — enabling an event

Two one-line flips (both files say `← EVENT-DAY SWITCH` on the line), then deploy:

1. `src/App.jsx` → `EVENTS["crunch-aloma"].active: false → true` (shows the page)
2. `api/_stripe.js` → `EVENT_SOURCES["crunch-aloma"].enabled: false → true` (grants the 30-day trial)

Deploy = commit + push main. After the event, flip both back and deploy again —
existing trial subscriptions keep their full 30 days; only NEW checkouts stop
getting the offer.

They're split on purpose (client shows, server grants) so a crafted API call can't
redeem a hidden offer, but that means **both must flip together** or you get a
visible page whose checkout returns "This offer isn't live yet."

## Adding a location

1. Add an entry to `EVENTS` in `src/App.jsx` (copy the aloma block; set path/copy,
   `active:false`).
2. Add the matching key to `EVENT_SOURCES` in `api/_stripe.js` (`enabled:false`).
3. Add `{ slug, url }` to `scripts/generate-qr.mjs` and run it; print the QR.

## Testing

- `node --env-file=.env scripts/test-event-trial.mjs` — end-to-end Stripe test-clock
  run of the 30-day trial: signup → `trialing` → clock past trial end → saved card
  auto-charged. Refuses to run against a live key.
- `npm run test:smoke` — includes the install-prompt specs.
- Wallet buttons (Apple Pay / Google Pay in the PaymentElement) require the payment
  method domain registration + wallet display prefs in the Stripe Dashboard (done
  for test mode 2026-07-02 via API; live needs the dashboard) and only show on real
  devices: Apple Pay in Safari on iOS, Google Pay in Chrome on Android.
  `public/.well-known/apple-developer-merchantid-domain-association` must stay
  deployed or live Apple Pay domain verification breaks.
